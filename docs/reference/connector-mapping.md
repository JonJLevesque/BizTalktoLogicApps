# BizTalk Adapter to Logic Apps Connector Mapping

> **Purpose**: Complete reference for migrating BizTalk adapter configurations to Logic Apps connector configurations.
> **Last updated**: 2026-02-23

---

## 1. Overview

BizTalk adapters = Logic Apps connectors. The mapping determines:
1. Which Logic Apps connector to use (built-in ServiceProvider vs managed ApiConnection)
2. How to translate the adapter's TransportTypeData to connector configuration
3. Whether an on-premises data gateway is required
4. Authentication migration approach

### Connector Type Decision

```
Is the connector available as Built-in (ServiceProvider)?
  YES → Use Built-in (in-process, better performance, simpler config)
  NO  → Use Managed (ApiConnection)
      → Is the target accessible only on-premises / private network?
         YES → Add on-premises data gateway
         NO  → Standard managed connector
```

## 3. Detailed Configuration Migration

### FILE Adapter → Azure Blob Storage

BizTalk BindingInfo.xml:
```xml
<Address>C:\Input\Orders\*.xml</Address>
<TransportTypeData>
  <![CDATA[<CustomProps>
    <PollingInterval vt="3">60000</PollingInterval>
    <FileMask vt="8">*.xml</FileMask>
    <BatchSize vt="3">20</BatchSize>
  </CustomProps>]]>
</TransportTypeData>
```

Logic Apps trigger configuration:
```json
"Poll_For_Orders": {
  "type": "ServiceProvider",
  "inputs": {
    "parameters": {
      "containerName": "orders-inbound",
      "blobMatchingCondition": { "matchWildcardPattern": "*.xml" }
    },
    "serviceProviderConfiguration": {
      "connectionName": "azureblob",
      "operationId": "getBlob",
      "serviceProviderId": "/serviceProviders/AzureBlob"
    }
  },
  "recurrence": { "frequency": "Minute", "interval": 1 }
}
```

**Translation rules**:
- `PollingInterval` (ms) → `recurrence.interval` + `recurrence.frequency` (Minute: 60000ms → interval:1, Minute)
- `FileMask` → `blobMatchingCondition.matchWildcardPattern`
- `C:\Input\Folder\` → Blob container name (folder path becomes container/virtual-folder)
- `%SourceFileName%` in send path → `@{triggerBody()?['Name']}` in output

### Service Bus (SB-Messaging) Adapter → Azure Service Bus Built-in

BizTalk BindingInfo.xml:
```xml
<Address>sb://myns.servicebus.windows.net/orders-queue</Address>
<TransportTypeData>
  <![CDATA[<CustomProps>
    <InboundTransportLocation>sb://myns.servicebus.windows.net/orders-queue</InboundTransportLocation>
    <SasKeyName>RootManageSharedAccessKey</SasKeyName>
    <SasKey>...</SasKey>
  </CustomProps>]]>
</TransportTypeData>
```

Logic Apps trigger:
```json
"When_SB_Message_Received": {
  "type": "ServiceProvider",
  "inputs": {
    "parameters": {
      "entityName": "orders-queue",
      "messageCount": 1,
      "receiveMode": "peekLock"
    },
    "serviceProviderConfiguration": {
      "connectionName": "serviceBus",
      "operationId": "receiveQueueMessages",
      "serviceProviderId": "/serviceProviders/serviceBus"
    }
  },
  "recurrence": { "frequency": "Second", "interval": 30 }
}
```

connections.json:
```json
"serviceBus": {
  "parameterValues": {
    "connectionString": "@appsetting('SERVICE_BUS_CONNECTION_STRING')"
  },
  "serviceProvider": { "id": "/serviceProviders/serviceBus" }
}
```

### WCF-BasicHttp Adapter → HTTP Action

BizTalk send port address: `http://erp.example.com/OrderService/OrderService.svc`

Logic Apps HTTP action:
```json
"Call_ERP_Service": {
  "type": "Http",
  "inputs": {
    "method": "POST",
    "uri": "http://erp.example.com/OrderService/OrderService.svc",
    "headers": {
      "Content-Type": "text/xml; charset=utf-8",
      "SOAPAction": "http://erp.example.com/IOrderService/SubmitOrder"
    },
    "body": "@{body('Build_SOAP_Envelope')}"
  }
}
```

### SQL Adapter → SQL Server Built-in

BizTalk SQL adapter (stored procedure trigger):
```
PollingStatement = EXEC dbo.GetPendingOrders
```

Logic Apps recurrence + SQL:
```json
"Poll_Database": {
  "type": "Recurrence",
  "recurrence": { "frequency": "Minute", "interval": 5 }
},
"Get_Pending_Orders": {
  "type": "ServiceProvider",
  "inputs": {
    "parameters": {
      "query": "EXEC dbo.GetPendingOrders"
    },
    "serviceProviderConfiguration": {
      "connectionName": "sql",
      "operationId": "executeQuery",
      "serviceProviderId": "/serviceProviders/sql"
    }
  },
  "runAfter": { "Poll_Database": [] }
}
```

## 4. Authentication Migration

| BizTalk Authentication | Logic Apps Equivalent |
|---|---|
| Windows Integrated (NTLM/Kerberos) | Managed Identity (preferred) or service account credentials |
| X.509 client certificate | `"authentication": {"type": "ClientCertificate"}` in HTTP action |
| Basic auth (username/password) | `"authentication": {"type": "Basic"}` — store in Key Vault |
| OAuth 2.0 | `"authentication": {"type": "ActiveDirectoryOAuth"}` |
| SSO affiliate application | Azure Key Vault secret reference: `@appsetting('SECRET_NAME')` |
| WCF transport security (certificate) | HTTP + client cert auth |
| WCF message security | HTTP + SOAP headers (manual) or Azure Function |
| No authentication | None |
| API Key (header) | `"headers": {"api-key": "@appsetting('API_KEY')"}` |
| SAS token (Service Bus/Event Hubs) | Connection string in app setting → Key Vault reference |

## 5. On-Premises Connectivity

For adapters connecting to on-premises systems, Logic Apps uses:

| Option | Use Case | Setup |
|---|---|---|
| **On-Premises Data Gateway** | SQL Server, SharePoint, File System, SAP (on-prem) | Install gateway on on-prem server; register in Azure |
| **Azure Relay** | WCF relay, hybrid HTTP | Azure Relay namespace; Hybrid Connection |
| **VNET Integration** | Private network access for Standard | Standard hosting plan with VNET injection |
| **ExpressRoute / VPN** | Full network connectivity | Azure networking infrastructure |
| **Azure Arc** | Logic Apps on-prem with Hybrid SKU | Deploy Logic Apps Standard in Container Apps extension |

### Adapter On-Prem Requirements

| Adapter | On-Prem Connectivity Needed |
|---|---|
| FILE (on-prem folder) | On-premises data gateway (FileSystem connector) |
| SQL Server (on-prem) | On-premises data gateway (SQL connector) |
| SharePoint (on-prem) | On-premises data gateway (SharePoint connector) |
| SAP (on-prem) | On-premises data gateway (SAP connector) |
| WCF-NetTcp | Azure Relay or Azure Function on-prem |
| MSMQ | Service Bus migration recommended; or on-prem gateway |
| FILE (Azure Blob) | No gateway needed (public cloud) |
| HTTP/HTTPS | No gateway if target is internet-accessible |

## 6. Migration Decision Matrix

Use this to determine the correct Logic Apps pattern for each BizTalk adapter:

```
Is the target system in Azure or on the internet?
  YES → Use built-in ServiceProvider connector (if available) or standard HTTP
  NO  → Is it on-premises?
    YES → Does Logic Apps Standard have a built-in connector for it?
      YES (SQL, File, SharePoint, SAP) → Built-in connector + on-premises gateway
      NO → Options:
        a. Azure Function (wraps the on-prem call)
        b. Azure Relay (for WCF/HTTP scenarios)
        c. Redesign endpoint to expose HTTP API
    NO → Check managed connector catalog (1,400+ managed connectors)
      FOUND → Use managed ApiConnection connector
      NOT FOUND → Azure Functions custom HTTP wrapper
```

### Performance Recommendations

1. **Prefer Built-in over Managed** when available (in-process, no extra hop)
2. **Service Bus for high-throughput** replacing MSMQ/WCF-NetMsmq
3. **Blob for file patterns** replacing FILE adapter (better retry, event-driven)
4. **HTTP action** for one-off WCF-BasicHttp/WCF-WSHttp calls (simple, no connector overhead)
5. **Azure Functions** for protocol adapters with no connector equivalent (WCF-NetTcp, custom pipeline components)
