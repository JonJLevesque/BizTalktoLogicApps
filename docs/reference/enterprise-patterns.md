# Enterprise Integration Patterns — BizTalk to Logic Apps Coverage

> Reference: [Microsoft Learn — Implementing Design Patterns in Orchestrations](https://learn.microsoft.com/en-us/biztalk/core/implementing-design-patterns-in-orchestrations)
> Source: Hohpe & Woolf, *Enterprise Integration Patterns* (2003)

This document maps each BizTalk-documented enterprise integration pattern to its Logic Apps
equivalent, support level, and estimated migration effort.

---

## Support Levels

| Symbol | Level | Meaning |
|--------|-------|---------|
| ✅ Auto | Automatic | Migrates automatically — tool generates the correct Logic Apps construct |
| ⚠️ Partial | Partial | Migrates with caveats — some manual review or redesign of specific sub-components |
| ❌ Redesign | Redesign required | No direct Logic Apps equivalent — architectural redesign needed |

## Pattern Details

### ✅ Content-Based Router

**BizTalk implementation:** `DecisionShape` (2 branches → `If`) or `Switch` construct (N branches → `Switch`).

**Logic Apps equivalent:**
- 2 branches: `If` action with JSON predicate expression
- N branches: `Switch` action with named cases

**Notes:** XLANG/s conditions are translated directly using the Section 4 expression table.
The tool handles `==`, `!=`, `>`, `<`, `&&`, `||` operators.
C# `.NET` methods in conditions (e.g. `Regex.Match`) require Azure Functions — flagged in gap analysis.

---

### ✅ Message Translator

**BizTalk implementation:** `TransformShape` using a `.btm` BizTalk Map.

**Logic Apps equivalent:**
- `.btm` with no C# scripting → `Xslt` action (Integration Account) — converted to XSLT
- `.btm` with C# scripting functoids → `Xslt` action with Azure Functions replacements
- Simple field-to-field maps → Data Mapper LML (no Integration Account needed)

**Notes:** C# scripting functoids (`msxsl:script` blocks) are NOT supported in Logic Apps XSLT.
The tool generates the XSLT and flags any `<Script>` blocks in the gap analysis.
Integration Account required for XSLT execution.

---

### ✅ Message Filter

**BizTalk implementation:** Send port filter expressions on promoted properties.

**Logic Apps equivalent:** `Condition` action before the send, with a `Terminate` action on the
"false" branch (discard path).

**Notes:** BizTalk MessageBox subscriptions that filter by promoted properties are re-implemented
as inline conditions. Multiple subscriber ports (pub-sub) are handled separately (see Message Broker).

---

### ✅ Request-Reply

**BizTalk implementation:** Two-way orchestration port — activating `ReceiveShape` on an
`Implements` port + `SendShape` replying on the same port.

**Logic Apps equivalent:** `Request` trigger (webhook) + `Response` action.
The trigger URL is auto-generated by Logic Apps.

**Notes:** HTTP and WCF-BasicHttp map cleanly. WCF-WSHttp requires manual WS-Security header
configuration. WCF-NetTcp and WCF-NetNamedPipe require redesign (see Section 3 adapter table).

---

### ✅ Wire Tap

**BizTalk implementation:** Additional `SendShape` to an audit/logging port alongside the main
processing path — port name heuristics: "audit", "log", "archive", "track", "monitor", "copy".

**Logic Apps equivalent:** Additional ServiceProvider `Send` action running in parallel after
the same `runAfter` dependency (same predecessor action).

**Notes:** The tool detects wire tap by port name pattern matching. Rename your audit port to
include one of the trigger keywords if the detector misses it.

---

### ✅ Dead Letter Queue

**BizTalk implementation:** `ScopeShape` with exception handling that routes failed messages
to a dedicated dead-letter send port.

**Logic Apps equivalent:** `Scope` action wrapping the main processing steps.
A `Terminate` action (or dead-letter Service Bus send) in the error branch:
```json
"runAfter": { "Scope_Main": ["FAILED", "TIMEDOUT"] }
```

**Notes:** Service Bus built-in DLQ used for poison messages. Run History in Azure Portal
replaces BizTalk Admin Console suspended message queues.

---

### ✅ Claim Check

**BizTalk implementation:** Large message stored to FILE adapter; downstream orchestrations
reference the file path rather than the message body.

**Logic Apps equivalent:** Azure Blob Storage upload action (ServiceProvider `azureblob`)
followed by passing the blob URI/path as a lightweight reference through subsequent workflow steps.

---

### ✅ Message Enricher

**BizTalk implementation:** `ReceiveShape` (activating) → `SendShape` (lookup request) →
`ReceiveShape` (non-activating, correlated) → `ConstructShape` (build enriched message).

**Logic Apps equivalent:** Trigger → HTTP `Send` action (lookup call) → `Compose` action
(merge original body with lookup response fields).

**Notes:** The correlation set is replaced by stateful workflow state — Logic Apps Standard
stateful workflows preserve all in-flight data across HTTP call-outs automatically.

---

### ⚠️ Sequential Convoy

**BizTalk implementation:** Multiple correlated `ReceiveShapes` — messages arrive in sequence
and each must be processed before the next is accepted.

**Logic Apps equivalent:** Service Bus queue with **sessions enabled** (session ID = correlation key).
A trigger configured for a specific session ensures FIFO processing per session.

**Caveats:**
- Session ID must be set on all inbound messages (requires upstream coordination)
- Logic Apps Standard handles one session at a time per workflow instance
- Parallel session processing requires multiple workflow replicas

**Effort:** 2-3 days to configure Service Bus sessions + session-aware trigger + correlation mapping.

---

### ⚠️ Splitter

**BizTalk implementation:** Envelope debatching in receive pipeline (XML Disassembler or
EDI/AS2 pipeline) — one inbound batch → N individual messages to MessageBox.

**Logic Apps equivalent:** `ForEach` action with `concurrency: 1` for sequential processing.
Parse the batch XML/JSON first, then iterate over the array.

**Caveats:**
- Logic Apps `ForEach` default is parallel (all at once) — must set concurrency to 1 for
  ordered sequential processing to match BizTalk semantics
- Flat file debatching (FlatFile Disassembler) produces different XML than Logic Apps
  Flat File Decode action — manual schema alignment required
- EDI batch splitting requires Integration Account with EDI partner configuration

**Effort:** 1-2 days for standard XML; 3-5 days for EDI/flat file.

---

### ⚠️ Aggregator

**BizTalk implementation:** `LoopShape` accumulating correlated messages into a batch
until a completion condition is met.

**Logic Apps equivalent:** `ForEach` action accumulating results into a variable.
No native aggregation window (time-based or count-based batching not built-in).

**Caveats:**
- No native "collect N messages then fire" batching — must implement with a counter variable
- Time-window aggregation requires an external timer trigger (Recurrence) + Cosmos DB or
  Table Storage to accumulate messages between trigger invocations
- Logic Apps `Initialize Variable` + `Append to Array Variable` provides basic accumulation

**Effort:** 2-5 days depending on aggregation window complexity.

---

### ⚠️ Scatter-Gather

**BizTalk implementation:** `ParallelActionsShape` with multiple `SendShapes` (scatter) +
subsequent aggregation step collecting all responses (gather).

**Logic Apps equivalent:** Multiple actions sharing the same `runAfter` dependency run in
parallel. A subsequent `ForEach` collects all responses.

**Caveats:**
- Logic Apps does not natively "wait for all parallel actions" with individual timeouts
- All parallel branches must complete before the gather step (standard `runAfter` handles this)
- Dynamic scatter (variable number of targets at runtime) requires a `ForEach` for the send phase

**Effort:** 3-5 days for static scatter; 5-7 days for dynamic scatter.

---

### ⚠️ Composed Message Processor

**BizTalk implementation:** Split incoming batch → route each sub-message via CBR → process
individually → re-aggregate results.

**Logic Apps equivalent:** `ForEach` (split) → nested `Switch` or `If` (route) → individual
processing actions → outer `Append to Array Variable` (aggregate).

**Caveats:** Combines Splitter + Content-Based Router + Aggregator — all caveats above apply.
The re-aggregation step is the most complex to implement correctly.

**Effort:** 3-5 days (assumes components are individually straightforward).

---

### ⚠️ Parallel Convoy

**BizTalk implementation:** Multiple simultaneous correlated receives from different sources,
all needed before processing continues (AND-join semantics).

**Logic Apps equivalent:** Multiple Service Bus subscriptions (one per message source) → each
triggers a separate workflow → all write to shared Cosmos DB or Table Storage → a third
workflow polls until all expected messages have arrived (AND-join check).

**Caveats:**
- No native AND-join in Logic Apps — requires external state store for tracking arrival
- First-event-wins semantics are easier: Event Grid + Logic Apps handles this
- Complex correlation scenarios may require Durable Functions instead

**Effort:** 3-7 days depending on number of concurrent paths and correlation complexity.

---

### ⚠️ Suspend with Retry

**BizTalk implementation:** `SuspendShape` pauses orchestration waiting for admin intervention,
then resumes from where it stopped. Often combined with a retry counter.

**Logic Apps equivalent:**
- **Retry only:** `Until` loop with counter variable — tool generates this automatically
- **Suspend + manual resume:** `Terminate` (Failed) → re-trigger via Service Bus DLQ + separate
  resume workflow — no native Suspend equivalent in Logic Apps

**Caveats:**
- Logic Apps has no native Suspend state — cannot pause mid-workflow and resume in-place
- Retry loops (no suspend) migrate automatically
- Manual intervention workflows must be redesigned as "fail-and-resubmit" patterns

**Effort:** 2-4 days (retry only: < 1 day; suspend+resume redesign: 3-4 days).

---

### ⚠️ Process Manager

**BizTalk implementation:** Long-running orchestration managing a multi-step business process
with multiple state transitions, correlation sets, and sub-orchestration calls.

**Logic Apps equivalent:** Stateful workflow (always used for BizTalk migration) +
child `Workflow` actions for sub-orchestrations. Correlation handled by stateful execution context.

**Caveats:**
- Complex correlation sets require careful mapping of session/instance IDs
- Long-running processes (days/weeks) work well in Logic Apps Standard stateful workflows
- BizTalk Orchestration Designer diagrams do not have a visual equivalent in Logic Apps Designer

**Effort:** 3-7 days depending on number of states and sub-orchestrations.

---

### ❌ Message Broker (MessageBox Publish-Subscribe)

**BizTalk implementation:** Every message is published to the MessageBox. Send ports with
filter subscriptions receive matching messages — full pub-sub at the infrastructure level.

**Logic Apps equivalent:** Service Bus Topics (one per message type) + Event Grid (for
event-driven pub-sub). Each subscriber needs its own Logic App workflow.

**Why it requires redesign:**
- BizTalk MessageBox is a central routing fabric — Logic Apps has no equivalent infrastructure
- Each BizTalk send port subscription must be re-implemented as a separate Logic App workflow
  with its own trigger (Service Bus subscription or Event Grid subscription)
- Promoted property routing must be re-implemented as message header routing or topic filters

**Effort:** 5-10 days for a typical pub-sub application; scales with number of subscriber ports.

---

### ❌ Compensation

**BizTalk implementation:** `CompensateShape` inside an `AtomicTransactionScope` —
undoes completed work in a distributed transaction on failure. Uses MSDTC.

**Logic Apps equivalent:** Saga pattern — explicit compensation workflows that undo each
completed step on failure. No 2-phase commit in Azure.

**Why it requires redesign:**
- MSDTC is not available in Azure
- Each transactional step must be idempotent or have an explicit undo action
- Outbox pattern recommended for DB + messaging scenarios

**Effort:** 5-15 days depending on the number of transactional resources and undo complexity.

## Summary

| Category | Patterns |
|----------|---------|
| **Migrate automatically (✅ Auto)** | Content-Based Router, Message Translator, Message Filter, Request-Reply, Wire Tap, Dead Letter Queue, Claim Check, Message Enricher |
| **Migrate with caveats (⚠️ Partial)** | Sequential Convoy, Splitter, Aggregator, Scatter-Gather, Composed Message Processor, Parallel Convoy, Suspend with Retry, Process Manager |
| **Require redesign (❌)** | Message Broker, Compensation |

**8 of 18 patterns migrate automatically.
8 of 18 migrate with caveats (review required).
2 of 18 require full architectural redesign.**
